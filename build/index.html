Introduction

As a member of the front-end development team you will be expected adhere to the standards and guidelines detailed within this document. Some of the information within relates to how we in the Miami Studio will develop our sites, but most will concern the idea of best practices. A 'Best Practice' is defined as:

'a method or technique that has consistently shown results superior to those achieved with other means'

They are not rules. Ultimately each developer is expected to use their best judgement when applying the guidelines listed in this document.

The information here has been collected from the experiences we've had with numerous, large‐scale projects, as well as a variety of web resources. These standards will help reduce friction in your workflows by providing a common starting point and will serve as a useful intro for both new hires and contractors.

General Guidelines

Client Requirements

Occasionally a client may have a requirement or best practice that conflicts with the ones contained in this document. For example, the client would like us to use camel case instead of hyphen delimiters to name css classes. It's times like these when the old saying "The client is always right" comes into play. Unless you feel that the request could jeopardize the delivery of the project any client requirements take precedence over the ones outlined in this document.

Always ask for client requirements before you start coding. In most cases they wont have any, but if they do, you and your team should revise them before you start coding and make sure they are followed throughout the life of the project.

Version Control

All Sapient projects should use source control. We generally have a SVN repository setup through ResultSpace3, some projects may use a different version control system depending on client requirements. All members of a team must have access to version control and are mandated to use it.

Do not begin development without access to a version control repository!

Automation

Whether it's compiling LESS files into CSS or linting your JS files automation plays a major role in development here. Grunt.js is our tool of choice for automation tasks.

Before any project ends up going on a production server the following tasks should be run on your code :

  1. lint js
  2. concat js files
  3. minify js files
  4. compile LESS files to CSS
  5. lint css
  6. concat css files
  7. minify css files
  8. lint html

Including Files

Including CSS

For including CSS use <link> and not @import as the standard. This is an important performance consideration. CSS files should always be included in the HEAD of the document. Placing style sheets in the head allows the page to render progressively.

For more information on CSS please see our CSS Standards.

Including JS

JS files should always be included near the END of a document, unless there is specific reason for importing them elsewhere. Keep the number of JS files to a minimum, join them whenever possible. Avoid inline‐JS unless you have a good reason to use it.

For more information on Javascript please see our Javascript Standards.

Don't Repeat Yourself

Don't repeat yourself (DRY) when writing code. Try to chunk different operations and processes into separate functions, but don't overdo it. Always think about maintainability and readability. Before copying and pasting code consider creating a variable, a function or a seprate file to be loaded in.

Reducing the Number of HTTP Requests

One of the crucial things when it comes to improving performance is reducing the number of HTTP requests. The latency between HTTP requests, and also the limitation in web browsers in regards to how many parallel HTTP requests a web browser can make, vastly affects the loading time of a web page. The reduction of HTTP requests is especially important on mobile platforms.

Javascript and CSS

Files should be concatenated according to usage and relevance. You don't want to unnecessarily load code you won't be using but you also do not want to make multiple HTTP Requests for code that could be merged into one file.

CSS Sprites

CSS Sprites result from the combination of a number of images into the same image file. You then use background-position in the CSS file to control what part of that image will be visible for a certain element. This way, instead of having a number of HTTP requests, there will be just one for a larger image file.

This practice works best for image elements that repeat across a site.

Indentation

Code should be indented using spaces rather than tabs.

Use indentation consistently to enhance the readability of the code. When elements carry over more than one line of code, indent the contents of elements between the start tag and the end tag. This will make it easy to see where the element begins and ends. Also, use indentation to align code at attribute names. Avoid over-nesting HTML. The simpler the markup the better.

Use four spaces for each level in indentation. Choose a development tool that allows you to convert tabs to spaces so that the indentation will be the same when the source is viewed in different editors or as printed output.

HTML

HTML(5)

Whenever possible we write HTML5 code, unless there is a very specific need to use XHTML. When you write any HTML, it has to be:

  • Well‐formed and generally valid (You must use a validator!). No funky nesting.
  • Slim ‐ no unnecessary bloating or extra containers around containers around containers.
  • Separated from presentation (CSS) and behavior (JS).
  • Implemented using the right tags otherwise know as semantics.

Semantics

The concept of semantics goes hand in hand with HTML5. Review the PDF included below, it will help you choose what element is most appropriate for your content.

For more information on specific tags and their appropriate use visit HTML5 Doctor | Element Index.

The <html> tag

The HTML5 Boilerplate uses conditional comments to dynamically assign classes to the <html> tag. 

<!--[if lt IE 7]>
<html class="no-js lt-ie9 lt-ie8 lt-ie7" lang="en"> <![endif]-->
<!--[if IE 7]>
<html class="no-js lt-ie9 lt-ie8" lang="en"> <![endif]-->
<!--[if IE 8]>
<html class="no-js lt-ie9" lang="en"> <![endif]-->
<!--[if gt IE 8]><!--> <html class="no-js" lang="en"> <!--<![endif]-->

The <head> tag

The text in these tags is not displayed, but parsable and tells the browsers (or other web services) specific information about the page. Simply, it “explains” the page so a browser can understand it.

Meta tags, style sheets, page title and occasionally javascripts are found within the <head> tag.

Character Encoding

Make sure UTF‐8 is used to encode your files unless you have a specific reason to use something else. Declare it by placing this code in the first line of your head tag.

<head>
    <meta charset="utf-8">

Unless you chose XHTML then you have to use:

<head>
    <meta http-equiv="Content-Type" content="text/html; charset=UTF-8" />

Mobile

Meta Tags for Mobile

<meta name="format-detection" content="telephone=yes">
<meta name="format-detection" content="telephone=no">

This tag determines whether or not telephone numbers in the HTML content will appear as hypertext links. The user can click a link with a telephone number to initiate a phone call to that phone number.

<meta name="HandheldFriendly">

This tag indicates to the browser that the page has been laid out by the developer to be viewable on a small screen, and that the browser should not attempt any further layout enhancements. While somewhat effective, it does not offer any granular control over the layout width ‐ it’s just a true/false flag.

viewport
<meta name="viewport" content="width=device-width,initial-scale=1,maximum-scale=1">

Of the meta tags specific to mobile this one is the most important. This meta tag can be used to control how HTML content will appear in mobile browsers.

Available options include:

Property Option
width width=device‐width
width=### ( ### is the number of pixels between 200 and 10000 ex. width=980 )
height height=device‐height
height=### ( ### is the number of pixels between 223 and 10000 ex. height=980 )
minimum‐scale minimum‐scale=#.## ( #.## is a floating point number between 0.0 and 10.0 ex. minimum‐scale=0.25 )
maximum‐scale maximum‐scale=#.## ( #.## is a floating point number between 0.0 and 10.0 ex. maximum‐scale=1.6 )
initial‐scale initial‐scale=#.## ( #.## is a floating point number between minimum‐scale and maximum‐scale )
user‐scalable user‐scalable=yes ( yes by default )
user‐scalable=no

Targeting Screen Sizes

<link rel="stylesheet" media="only screen and (max-device-width: 320px)" href="phone.css">
<link rel="stylesheet" media="only screen and (min-width: 641px) and (max-width: 800px)" href="ipad.css">

Apple specific tags

Although these tags originated with Apple others have adopted them as well. Any site targeting mobile specifically should consider including the follow snippets within the 'head' tag of the page.

<link rel="apple-touch-icon" href="apple-touch-icon.png">
<!-- For non-Retina iPhone, iPod Touch, and Android 2.1+ devices: -->
<link rel="apple-touch-icon-precomposed" href="apple-touch-iconprecomposed.png">
<!-- For first- and second-generation iPad: -->
<link rel="apple-touch-icon-precomposed" sizes="72x72" href="apple-touchicon-72x72-precomposed.png">
<!-- For iPhone with high-resolution Retina display: -->
<link rel="apple-touch-icon-precomposed" sizes="114x114" href="appletouch-icon-114x114-precomposed.png">
<!-- For third-generation iPad with high-resolution Retina display: -->
<link rel="apple-touch-icon-precomposed" sizes="144x144" href="appletouch-icon-144x144-precomposed.png">

This tag sets the icon for a Web Clip. A Web Clip is a link to a web page or content from a web page that has been saved to a device along with an icon on the home screen that can be used to display it.

apple-touch-startup-image
<link rel="apple-touch-startup-image">

This tag specifies a startup image, which is an image that is displayed while an offline web application is being launched on the mobile device. For the iPhone and iPod Touch, the image must be in portrait orientation 320 pixels wide by 460 pixels high.

apple-mobile-web-app-capable
<meta name="apple-mobile-web-app-capable" content='yes'>
<meta name="apple-mobile-web-app-capable" content='no'>

For a mobile app, the apple‐mobile‐web‐app‐capable meta tag determines whether the application runs in web mode, with an address bar and navigation bar, or in full screen mode without them.

The 'content' attribute will determine if your page will display the address bar at the top and the navigation bar at the bottom of the screen. A value of 'yes' will hide the address and navigation bars, 'no' will show them.

apple-mobile-web-app-status-bar-style
<meta name="apple-mobile-web-app-status-bar-style" content='default'>
<meta name="apple-mobile-web-app-status-bar-style" content='black'>
<meta name="apple-mobile-web-app-status-bar-style" content='black-translucent'>

This meta tag has no effect unless you first specify full-screen mode as described in 'apple-mobile-web-app-capable.'

  • default - The status bar color is the normal gray gradient and it does not overlap the HTML content.
  • black - The status bar is black and does not overlap the HTML content.
  • black-translucent - The status bar is translucent black and overlays a small strip at the top of the HTML content.

CSS

How to write good CSS (Selectors)

Writing Efficient CSS selectors is a good article to understand why writing efficient CSS selectors is important. Here's a summary:

  • Keep specificity low (that is, don't write html body .wrapper #content a{} when .content a{} will accomplish the same
  • Try to avoid putting unnecessary classes and IDs in your HTML
  • The fewer matches your key selector (the one to the very right hand side of your selector) has, the faster your selector will be overall.
  • Avoid the use of IDs this limits the reusability of your styles and can create maintenance headaches.
  • Avoid the * selector unless you can explain why you MUST use it.
  • Avoid the !important unless you can explain why you MUST use it.

There is much to be said around how to architect the HTML and subsequently the CSS for a larger site. Here are a few good pointers to keep in mind:

  1. Find patterns in different modules and abstract them out.

    e.g. Two modules that have bulleted lists in them might inherit from a common class, and be adjusted separately with additional rules.

  2. When you create a module, let it be flexible so it can be used in a different context

    This goes hand in hand with "keep specificity low". When you write CSS for any module, don't tie it to a specific place (or parent) unless you really have to.

Selector Efficiency

In terms of performance, memorize this list of efficient CSS selectors ‐ ordered from higher to lower efficiency:

  1. ID, e.g. #header
  2. Class, e.g. .promo
  3. Type, e.g. div
  4. Adjacent sibling, e.g. h2 + p
  5. Child, e.g. li > ul
  6. Descendant, e.g. ul a
  7. Universal, i.e. *
  8. Attribute, e.g. input[type="text"]
  9. Pseudo‐classes/‐elements, e.g. a:hover

Use clear and thoughtful names for HTML classes. Pick an understandable and consistent naming pattern that makes sense both within HTML files and CSS files.

Semi-Colons

Always use a trailing semi-colon on the last declaration in a ruleset to avoid any potential confusion and syntax errors over the life of the document.

Formatting

The chosen code format must ensure that code:

  • is easy to read
  • is easy to clearly comment
  • minimizes the chance of accidentally introducing errors
  • results in useful diffs and blames.

To that end the following recommendations are made for CSS formatting:

  • One discrete selector per line in multi‐selector rulesets.
  • A single space before the opening brace of a ruleset.
  • One declaration per line in a declaration block.
  • One level of indentation for each declaration.
  • A single space after the colon of a declaration.
  • Always include a semicolon at the end of the last declaration in a declaration block.
  • Place the closing brace of a ruleset in the same column as the first character of the ruleset.
  • Separate each ruleset by a blank line.
  • Seperating words in a class name or id only with "‐"
  • Rules with various vendor‐prefixes should always be next to each other
.selector-1,
.selector-2,
.selector-3 {
    -webkit-box-sizing: border-box;
    -moz-box-sizing: border-box;
    box-sizing: border-box;
    display: block;
    color: #333;
    background: #fff;
}

Declaration order

Declarations should be ordered in accordance with a single principle, the team should initial agree to one of the suggested. Preference is for related properties to be grouped together and for structurally important properties (e.g. positioning and box‐model) to be declared prior to typographic, background, and color properties.

.selector {
    position: relative;
    display: block;
    width: 50%;
    height: 100px;
    padding: 10px;
    border: 0;
    margin: 10px;
    font-family: monospace;
    font-size:1em;
    color: #fff
    background: #000;
}

Do not sort declarations alphabetically, this approach separates related properties. For example, position offsets are no longer grouped together and box‐model properties can end up spread throughout a declaration block.

Exceptions and slight deviations

Large blocks of single declarations can use a slightly different, single‐line format. In this case, a space should be included after the opening brace and before the closing brace.

.selector-1 { width: 10%; }
.selector-2 { width: 20%; }
.selector-3 { width: 30%; }

Long, comma separated property values ‐ such as collections of gradients or shadows ‐ can be arranged across multiple lines in an effort to improve readability and produce more useful diffs. There are various formats that could be used; one example is shown below.

.selector {
    box-shadow:
        1px 1px 1px #000,
        2px 2px 1px 1px #ccc inset;
    background-image:
        linear-gradient(#fff, #ccc),
        linear-gradient(#f3c, #4ec);
}

Web Fonts

When using web fonts check if they are available on Google Web Fonts before generating the various formats necessary for the web.

If your font is not available on Google Web Fonts generate a font-face kit using FontSquirrel. Place your generated fonts in the css/fonts/ folder. Avoid including any additional css files and incorporate the necessary @font-face tags into an existing stylesheet.

Miscellaneous Standards

Avoid specifying units for zero‐values, e.g., margin: 0

Use lowercase hex values, e.g., #aaa

Use single or double quotes consistently. Preference is for double quotes, e.g., content: ""

Always quote attribute values in selectors, e.g., input[type="checkout"]

LESS

What is LESS?

CSS Preprocessors extend CSS with dynamic behavior such as variables, mixins, operations and functions. Our preprocessor of choice is LESS, for more information click here.

Naming Conventions

All applicable CSS naming conventions should be followed in your LESS implementation.

Use camel case when naming variables:

@tooltipColor: #fff;
@tooltipBackground: #000;
@tooltipArrowWidth: 5px;
@tooltipArrowColor: @tooltipBackground;

File Structure

There are five .less files that are included in our default templates:

  • normalize.less
  • variables.less
  • mixins.less
  • base.less
  • project.less
variables.less

All variables to be used within your LESS implementation should be defined in this file. Declarations should be in the following order :

  • Colors
  • Background/ Text Colors
  • Link Colors
  • Typography
  • Project Specific Variables
// -------------------------
// Variables
// -------------------------

// Greys
// -------------------------
@grey1: darken(#fff, 10%);
@grey2: darken(@grey1, 10%);
...

// Accent colors
// -------------------------
@blue:  #049cdb;
@green: #46a546;
@red: #9d261d;
...

// Scaffolding
// -------------------------
@bodyBackground: @white;
@textColor: @grey8;

// Links
// -------------------------
@linkColor: darken(@red, 10%);
@linkColorHover: darken(@linkColor, 15%);
@linkColorVisited: darken(@linkColor, 25%);

// Typography
// -------------------------
@sansFontFamily: "Helvetica Neue", Helvetica, Arial, sans-serif;
@serifFontFamily: Georgia, "Times New Roman", Times, serif;
@monoFontFamily: Monaco, Menlo, Consolas, "Courier New", monospace;

// Project Specific
// -------------------------
...
base.less

This file is used to combine various LESS files. It provides should provide common styles and variables that will be used throughout your LESS implementation. Ensure that any styles included within base.less will be used throughout the site. By default base.less looks like this:

// CSS Reset
@import "normalize.less"; // Maintain consistency across browsers

// Core variables and mixins
@import "variables.less"; // Modify this for custom colors, font-sizes, etc
@import "mixins.less";
project.less
@import "base.less";

// Project specific code
...

By using this approach we can determine the order in which our LESS files will be combined.

Formatting

Mixins

Group mixins toward the top of your style declarations. This makes it easy for other developers to see what mixins are being applied. It also allows for values to be overridden if necessary.

section {
    .rounded-border(5px);

    border-color:#ddd;
}
.rounded-border(@radius) {
  border: 1px solid #999;
  -webkit-border-radius: @radius @radius @radius @radius;
  -moz-border-radius: @radius @radius @radius @radius;
  border-radius: @radius @radius @radius @radius;

JavaScript

General Guidelines

Separate Javascript Functionality from HTML

Separate Javascript functionality into a “behavioural layer,” so that it is separate from and independent of HTML and CSS. HTML is the markup, CSS the presentation and Javascript the behavioural layer. This means storing ALL Javascript code in external script files and building pages that do not rely on Javascript to be usable.

<!-- HTML -->
<a onclick="doSomething()" href="#">Click!</a>

<!-- HTML -->
<a href="backuplink.html" class="doSomething">Click!</a>

// Javascript
$('a.doSomething').click(function(){
    alert('You did something, woo hoo!');
});
Load scripts asynchronously

The less there is to initialize and to process in the beginning, the faster your website loads. This is not directly affecting your JavaScript performance, but at the end of the day the user only cares about the performance of your website.

Please note, that this is only relevant when it comes to a certain level of Javascript in size and complexity. For the majority of websites, it's usually recommend to save additional requests and combine & minify your JavaScript into one file as well as compress (gzip) it for delivery. This is also the easiest way to leverage caching mechanisms of browsers.

JSLint your code

Mr. Douglas Crockford, the father of JavaScript has some ideas about how "good" JS code looks like and he has written a JSLint script to validate any code against it. Although there might be good reasons to not follow each and every rule, it's usually a very good starting point for getting your code clean, stable and increasing the compression rate for delivery.

Avoid "eval"

The well‐known JavaScript rule "eval is evil" is there for a reason. You should avoid, unless you're 100% (better make it 1000%) sure, you are facing one of the rare exceptions in which it's appropriate.

The eval() function in JavaScript is a way to run arbitrary code at run‐time. That means you're running any (!) kind of code within the current scope and execution context. This violates the integrity of your variable values (identifier conflicts) and is a serious threat to the security of your code or even your entire website.

Naming Conventions

Variables

Identifiers (other than constructors) should start with a lowercase letter and use camelCase for multiple words.

var _myVar = 'foo';
var my_var = 'foo';
var MYVAR = 'foo';
var $myvar = 'foo';
function MyFunction() {
}
function MY_FUNCTION() {
}

var myVar = 'foo';
function myFunction() {
}
Constructors

Constructors should start with an uppercase letter and use camelCase for multiple words to better conform with native JavaScript syntax.

function foo() {};
foo.prototype = {};
new foo;

function Foo() {};
Foo.prototype = {};
new Foo;
Quotes

Use single quotes. Saves a keystroke and simplifies readability. Also, string literals more often contain single quotes than double quotes, so this reduces the amount of escaping necessary.

var foo = "bar";

var foo = 'bar';
Brackets

Place opening brackets on same line as preceding statement. This avoids errors in cases where automatic semi‐colon insertion causes the return statement to return undefined instead of an object literal.

function foo()
{
}
return
{
    "myMethod": myMethod
}

function foo() {
}
return {
    "myMethod": myMethod
}
Variable Declarations

Declare all vars at the top of their scope when possible. This helps to one understand the scope's variables and makes maintenance easier.

function sample(condition) {
    if ( condition ) {
     var foo = 'bar';
    }
    return foo;
}

function sample(condition) {
    var foo;
    if ( condition ) {
        foo = 'bar';
    }
    return foo;
}

When working with a large number of variables declare all vars with one var keyword when appropriate and use preceding commas. Although this method is more difficult to maintain it can be helpful when working with a large list variables.

var foo;
var bar;
var lorem;
var ipsum;
var dolor;

var foo,
    bar,
    lorem,
    ipsum,
    dolor;
Object Literals

Favor object literal notation for grouping object members. Grouping related members helps to keep code organized and allows editors to collapse blocks for readability.

var foo = {};
foo.prop = 'bar';
foo.method = function() {};

var foo = {
    prop: 'bar',
    method: function() {
    }
}

Performance

Reading/ Writing Values

The storing and accessing of values can differ in terms of speed depending of different methods. The impact of these different methods can make an impact when there are many read/write operations (real‐time calculations) versus storing large amounts of data (state caching). As a general note, try to use literals as often as possible. Especially when it comes to declaration of arrays/objects.

var array = new Array,
    object = new Object,
    string = new String;

var array = [ ],
    object = { },
    string = "";
Loops

In general the common "for" and "while" loops should be the preferred options. Apart from the loop‐type it's way more important to cut down the amount of operations that happen in each iteration.

A usual scenario is looping over an array, which can be optimized like this (maybe moving the var statement to the top of your function for readability and happy lint'ing):

// calculating the constant array-length in every iteration
for(var i = 0; i < arr.length; i++){
    do_stf();
}

// calculating the constant array-length once
for(var i = 0, len = arr.length; i < len; i++){
    do_stf();
}
// if the process order doesn't matter
// this performs only one operation per iteration
// and the loop stops at 0 (false) automatically
for(var i = arr.length ; i-- ; ){
    do_stf();
}
Strict Comparisions

Use strict comparisons when possible. Strict comparisons use the operand's value and type to determine the condition. This avoids type coercion (which has a performance cost), and more clearly communicates intent. For example, if a developer wrote if ( foo == null ) {...}, did he intend other false values (such as false, 0, '', or undefined) to pass this conditional?

if ( a == b )
if ( a != b )

if ( a === b )
if ( a !== b )
Avoid 'with'

A 'with' statement in JavaScript inserts an object at the front scope chain, so any property/variable references will first try to be resolved against this object. That means you add another level to the resolving process, which can impact your overall performance. Moreover conflicting identifier names can cause serious problems. Compare the following figure for with‐statement:

function add(num1,num2){
    with(document){
        var sum = num1 + num2,
        n = createTextNode(sum),
        p = getElementById(“t1”);
        p.appendChild(n);
        return sum;
    }
}
var total = add(5, 10);

jQuery

Best Practices

$(window).load()

There is a temptation among jQuery developers to hook everything into the $(document).ready pseudo event. After all, it is used in most examples you will find. Although $(document).ready is incredibly useful, it occurs during page render while objects are still downloading. If you notice your page stalling while loading, all those $(document).ready functions could be the reason why. You can reduce CPU utilization during the page load by binding your jQuery functions to the $(window).load event, which occurs after all objects called by the HTML (including <iframe> content) have downloaded. 

$(window).load(function(){
    // jQuery functions to initialize after the page has loaded.
});
Descend From an #id

The fastest selector in jQuery is the ID selector $('#someid'). This is because it maps directly to a native JavaScript method, getElementById(). Whenever possible descend from an id, especially when querying for multiple elements.

var buttons = $('.button');

var buttons = $('#content .button');
Use Tags Before Classes

The second fastest selector in jQuery is the Tag selector $('head'). Again, this is because it maps to a native JavaScript method, getElementsByTagName(). For additional efficiency include and id as well.

var content = $('input.on');
var content = $('#content input.on');
Use Sub-queries

jQuery allows us to run additional selector operations on a wrapped set. This reduces performance overhead on subsequent selections since we already grabbed and stored the parent object in a local variable.

var $traffic_light = $('#traffic_light'),
    $active_light = $traffic_light.find('input.on'),
    $inactive_lights = $traffic_light.find('input.off');
Cache query selector results

Way more important than finding the most efficient selector is caching your results for repeating tasks. Instead of traversing the same DOM tree branch over and over again, you should cache the results on initialization (if static) or after a related DOM update (i.e. after AJAX insert). Using jQuery here as an example, but the same is true for native JS or any other library/framework.

// traversing for "#myList" 2x on every click
$("#myButton").on('click', function (e) {
    if ($("#myList").hasClass("active")) {
        $("#myList").removeClass("active");
    } else {
        $("#myList").addClass("active");
    }
});

// caching the "#myList" Object once
var myList = $("#myList");
$("#myButton").on('click', function (e) {
    if (myList.hasClass("active")) {
        myList.removeClass("active");
    } else {
        myList.addClass("active");
    }
});
Chain multiple operations

Chaining operations in jQuery is more efficient than multiple operations against one object.

// instead of multiple operations on one HTML object ...
var myList = $("#myList");
myList.hide();
myList.css("background-color","#bada55");
myList.show();

// ... chain it and leverage built-in optimizations
$("#myList").hide()
    .css("background-color","#bada55")
    .show();
Limit Direct DOM Manipulation

The basic idea here is to create exactly what you need in memory, and then update the DOM. This is not a jQuery best practice, but a must for efficient JavaScript. Direct DOM manipulation is slow. 

var top_100_list = [...], // assume this has 100 unique strings
$mylist = $('#mylist'); // jQuery selects our <ul> element

for (var i=0, l=top_100_list.length; i<l; i++)
{
    $mylist.append('<li>' + top_100_list[i] + '</li>');
}

var top_100_list = [...], // assume this has 100 unique strings
$mylist = $('#mylist'), // jQuery selects our <ul> element
top_100_li = ""; // This will store our list items

for (var i=0, l=top_100_list.length; i<l; i++)
{
top_100_li += '<li>' + top_100_list[i] + '</li>';
}
$mylist.html(top_100_li);
Leverage Bubbling

Unless told otherwise, every event (e.g. click, mouseover, etc.) in JavaScript “bubbles” up the DOM tree to parent elements. This is incredibly useful when we want many elements (nodes) to call the same function. Instead of binding an event listener function to many nodes—very inefficient—you can bind it once to their parent, and have it figure out which node triggered the event.

$('#myList li).bind('click', function(){
    $(this).addClass('clicked');
    // do stuff
});

$('#myList).bind('click', function(e){
    var target = e.target, // e.target grabs the node that triggered the event.
    $target = $(target);  // wraps the node in a jQuery object
    if (target.nodeName === 'li') {
        $target.addClass('clicked');
        // do stuff
    }
});
